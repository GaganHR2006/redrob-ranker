Career history nested expander — potential StreamlitAPIException
CRITICAL  ·  Section 4.2 — Career History  ·  demo/app.py lines 215–217  ·  RISK / PARTIALLY OPEN
Steps
Load sample data (≤20 candidates) → Candidate Explorer tab → any candidate's Shadow Recruiter panel

For the ≤20 path, candidates are wrapped in st.container(border=True) — not st.expander. The inner career history expanders at lines 215–217 do NOT crash inside a container. However, the ≥21 candidate path (large sample upload) still uses with st.container(border=True) → show_candidate_evaluation() → inner expanders. This is safe. Suggestions for improvement
⚡
Cache scoring with st.cache_data (keyed on file hash)
Every Streamlit widget interaction triggers a full re-run, re-scoring all candidates. Wrapping the scoring loop with @st.cache_data keyed on hashlib.md5(content) eliminates redundant scoring when switching tabs or expanding panels. For 50 candidates this costs milliseconds; for larger uploads it prevents multi-second lags.
🗂️
Centralise a DISPLAY_MAPS dict for all status fields
BUG-04/05/06 were all the same class of bug: a raw signal value rendered without a display mapping. A single DISPLAY_MAPS = { "gem": {True:"✅", False:"❌"}, "deception": {"low":"🟢 Low",…}, "github": {-1:"Not linked"} } applied once before any DataFrame build prevents this class of bug entirely and makes future status fields easy to add.
📍
Define SAMPLE_DATA_PATH as a module-level constant
The hardcoded path inside the function body (BUG-01) is fragile and invisible. A single SAMPLE_DATA_PATH = Path(__file__).parent.parent / "sample_candidates.json" at the top of app.py makes the path visible, environment-portable, and unit-testable.
🔌
Wire cohort.py advanced panels into the Cohort tab
The cohort_report.json from the offline pipeline can be loaded and rendered as-is — zero re-computation required. Add four expandable sections at the bottom of show_cohort_analysis(): JD Coverage, Shared Gaps, Pairwise Tradeoffs, Anti-Bias Audit. This directly unlocks section 4.4 of the testing guide and substantially increases the demo's impressiveness for judges.
🛡️
Add lightweight schema validation on upload
The current upload handler catches json.JSONDecodeError but silently accepts structurally malformed arrays. A 4-line check verifying candidate_id, profile, career_history, redrob_signals are present on the first record would produce a clear error for edge-case uploads instead of a confusing empty table or KeyError traceback.
📊
Color-code the score distribution bars (red / grey / green zones)
The current Altair chart uses a single blue for all bars. Encoding color by score zone — red for 0.00–0.15 (keyword stuffers), grey for 0.15–0.50 (below threshold), green for 0.50+ (fits) — turns the chart into a visual story that immediately communicates the system's filtering logic to a judge without needing any caption.
